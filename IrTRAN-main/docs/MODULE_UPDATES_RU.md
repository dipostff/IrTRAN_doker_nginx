# Правки по модулям (сводка для разработки)

Документ описывает функциональные и структурные изменения, внесённые в клиентскую часть тренажёра (`projirgups/IrTRAN-main`). Цель — восстановить работоспособность форм, справочников подбора и сценариев обучения там, где интерфейс был статичным или не был связан с данными документа.

---

## 1. Акты: общая форма (ГУ-23) и коммерческий акт (ГУ-22)

### Проблема

Часть полей и таблиц не заполнялась, не было единых учебных справочников для подбора номеров актов, вагонов, отметок и связанных документов.

### Решение

| Компонент / файл | Назначение |
|------------------|------------|
| `src/config/actTrainingCatalogs.js` | Учебные каталоги: отметки отправителя, номера коммерческих актов, акты на начало простоя, демо-вагоны и контейнеры. При появлении API-справочников их можно подставить вместо статичных массивов. |
| `src/components/CommonActCreateComponent.vue` | Подключение подбора для персон, вагонов, отметок, документов; согласование с валидаторами обучения. |
| `src/components/CommercialActCreateComponent.vue` | Связка таблиц и полей с данными документа и каталогами. |
| `src/composables/useCommercialActForm.js` | Общая логика формы коммерческого акта (состояние, операции), чтобы не дублировать код в компоненте. |
| `src/helpers/trainingDocumentValidators.js` | Проверки для `common_act` и `commercial_act` при включённой «Проверке ошибок» в сценарии. |

### Зависимости данных

Станции, контрагенты и прочие сущности по возможности берутся из существующих вызовов API и Pinia (`useListsStore`), как в накладной и других модулях.

## 2. Памятка приёмосдатчика (форма ГУ-45)

### Проблема

Не работали таблица вагонов и модалка «Основной раздел памятки», вкладка «История», отображение номера документа до сохранения; в модалках возможен конфликт имени `document` (ref Vue vs `window.document`).

### Решение

| Область | Изменения |
|---------|-----------|
| **Данные документа** | Массив строк вагонов (`wagon_lines`), журнал `change_history`. |
| **Вкладки** | Управление через `activeTab` (Vue): «Документ» и «История»; без опоры только на `data-toggle` Bootstrap. |
| **Модалка вагона** | Черновик строки, кнопки «Применить» / закрытие; для DOM-операций используется **`window.document`**, чтобы не пересекаться с `ref` с именем `document`. |
| **Груз** | Подбор через API грузов (`getCargos`), отображение кода ЕТСНГ из справочника. |
| **Копирование / вставка** | Операции по выбранным строкам таблицы вагонов. |
| **Номер документа** | `assignNewDocumentIdIfNeeded()` — присвоение номера до первого сохранения для корректного заголовка. |
| **Редактирование строки вагона** | Обратный разбор `op_delivery` / `op_removal` (формат как у `formatDmHm`: `дд.мм` или `дд.мм чч:мм`) в поля даты/времени модалки (`parseDmHmToDateTime`, год из `createdAt` документа или текущий). При «Применить» поля, не редактируемые в модалке (`op_notify`, `delay_hm`, `gu23_act`, `weigh_count`, `note`, `railway_admin`, `wagon_ownership`), не затираются: в `rowFromWagonDraft` передаётся существующая строка при правке. |

**Файлы:** `src/components/ReminderCreateComponent.vue`, при необходимости правки в `ReminderCreateView.vue`.

## 3. Ведомость подачи и уборки

### Проблема

Форма была преимущественно вёрсткой: нет второй вкладки, поля без `v-model`, таблицы пустые, кнопки «Расчёт» без обработчиков, селекты без опций, часть полей с `disabled` без логики.

### Решение

#### Новые и изменённые файлы

| Файл | Назначение |
|------|------------|
| `src/config/fillingStatementCatalogs.js` | Тарифные планы, тип нумерации ведомости, принадлежность п/п, операции, нормы времени (в т.ч. вариант для кода «Ж» КОО-4), операции для срока оборота. |
| `src/helpers/fillingStatementCalculations.js` | `parseNum`, `fmtNum`, часы между `datetime-local`, расчёт производных сумм по коэффициентам, итог по блоку штрафа. |
| `src/components/FillingStatementCreateComponent.vue` | Полная модель документа, вкладки, таблицы, модалки, привязка блока «Штраф» и сборов. |

#### Расширенная модель `document` (ключевые поля)

- Шапка: `form2_number`, `tariff_plan_id`, `track_branch_id`, `statement_numbering_type`, даты/номер ведомости, период, часть полей оборота/неоплачиваемого времени.
- `cleaning_reminders[]` — памятки уборки (в т.ч. контрагент и станция затребования локомотива через `select-with-search`).
- `wagons_by_reminders[]` — вагоны: принадлежность (`ownerships`), группа (`rolling_stock_types`), груз и `code_ETSNG` из `cargos`, операции из каталога, расчётные часы в модалке.
- `fee_delivery_rows[]` — блок «Для расчета сборов за подачу и уборку».
- `delivery_summary`, `pp_usage_summary` — два блока с коэффициентами и производными суммами (подача/уборка и пользование ПП).
- `fine` — блок «Штраф» (начислено, платежи, коэффициенты, маневры, пробег, итого, сумма в уведомление).
- `change_history[]` — журнал; пополняется при сохранении и подписании.

#### Загрузка справочников

В `onMounted` дополнительно вызываются: `getCargos()`, `getRollingStockTypes()`, `getOwnerships()` (наряду со станциями, договорами, владельцами п/п, юрлицами).

#### Кнопки «Расчёт»

Реализованы учебные формулы (зависимость от часов вагонов, строк сборов, введённых коэффициентов и сумм). Они не заменяют промышленный расчёт АСУ и при подключении бэкенда должны быть заменены или делегированы API.

#### Ограничения

- Кнопки «Добавить из АСУ ЛР» остаются отключёнными без интеграции.
- «Извещение по сборам / по штрафам» в учебном режиме показывают информационное сообщение.

#### Нормализация сохранённых документов

Функция `normalizeFilling()` при открытии из `localStorage` дополняет отсутствующие поля и массивы значениями по умолчанию.

## 4. Общие замечания

- **Валидация обучения:** типы документов `common_act`, `commercial_act`, `reminder`, `filling_statement`, `cumulative_statement` проверяются в `trainingDocumentValidators.js` при активной проверке в сценарии.
- **Сборка:** после изменений рекомендуется `npm run build` в корне `IrTRAN-main`.
- **Дубликат кода:** при наличии копии проекта в `fail/projirgups` правки могли быть перенесены не полностью — сверять с `IrTRAN-main`.

## 5. Накопительная ведомость

### Проблема

Форма была частично статичной: секция «Сборы» не имела рабочей модели строк, графы в модалке не были связаны с данными документа, кнопки «Расчёт / Формула расчёта» не выполняли действий, поле «ж.д.» всегда пустое, «Итого к оплате» не пересчитывалось.

### Решение

| Файл | Назначение |
|------|------------|
| `src/components/CumulativeStatementCreateComponent.vue` | Привязка полей к данным документа, рабочий CRUD по строкам сборов, расчёты в модалке, пересчёт итога к оплате, вывод `ж.д.` из данных организации-перевозчика. |
| `src/config/cumulativeStatementCatalogs.js` | Учебные справочники: типы исходных документов, статьи сборов, дополнительные коды, варианты НДС. |

### Что теперь работает

- **`ж.д.`**: отображается из карточки выбранной организации (`railway` / `railway_code` / `road` / `short_name`).
- **`Сборы`**: таблица строк (`fee_rows`) с кнопками `Добавить/Изменить/Удалить/Копировать/Вставить`.
- **Графы модалки сбора**: `Номер документа`, `Статья сбора`, `Дополнительный код`, `Расчетная сумма`, `Ставка НДС`, `Сумма НДС`, суммы без коэффициентов и доходы от надбавок — все привязаны к модели строки.
- **`Расчёт`**: считает НДС и расчетную сумму (`Сумма сбора + НДС`), а также производные суммы по коэффициентам.
- **`Формула расчета`**: показывает используемую формулу пользователю.
- **`Итого к оплате`**: автоматически пересчитывается как сумма `calculated_amount` всех строк сборов.

### Примечание

Справочники для статей, доп.кодов и признаков НДС добавлены в учебном виде. При подключении целевого API их нужно заменить на загрузку из серверных справочников.
