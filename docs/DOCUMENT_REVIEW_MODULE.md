# Модуль «Проверка документов»

Документ описывает текущую реализацию модуля проверки документов в `projirgups` (MVP + расширения UI/аналитики).

## Цель модуля

Модуль позволяет:
- студенту отправить заполненный документ на проверку;
- преподавателю проверить документ вручную или с алгоритмической подсказкой;
- выставить результат (`Принято/Не принято`, оценка, комментарий, возможность переделки);
- отследить историю версий и переоценок.

## Роли и доступ

- `student`
  - отправка документа на проверку;
  - просмотр результата в «Успеваемости»;
  - повторная отправка при разрешённой переделке.
- `teacher`
  - просмотр отправок своих студентов (по группам Keycloak);
  - ручная/алгоритмическая проверка;
  - переоценка отправки.
- `app-admin`
  - полный доступ к отправкам и шаблонам (без ограничений группой).

## Пользовательский поток

1. Студент заполняет документ в учебном модуле.
2. В «Документы пользователя» нажимает «Отправить на проверку».
3. Создаётся запись в `document_reviews` со статусом `submitted` и `payload_snapshot`.
4. Преподаватель видит отправку в модуле «Проверка документов».
5. Преподаватель:
   - открывает отправку;
   - при необходимости открывает readonly-страницу документа;
   - проверяет чек-лист и блок проблем;
   - запускает алгоритм (опционально);
   - выставляет итог.
6. Статус меняется на `reviewed`, студент получает уведомление, результат попадает в «Успеваемость».

## Данные и модели

### `DocumentReview` (`document_reviews`)

Ключевые поля:
- идентификация: `document_source`, `document_id`, `document_type`, `student_user_id`, `version_no`;
- статус/сроки: `status`, `submitted_at`, `reviewed_at`;
- проверка: `check_mode`, `algorithm_summary`, `algorithm_recommendation`;
- итог преподавателя: `acceptance`, `grade`, `teacher_comment`, `can_rework`;
- снимок документа: `payload_snapshot`.

### `DocumentReviewTemplate` (`document_review_templates`)

Один шаблон на тип документа:
- `document_type`;
- `payload` (эталон);
- `updated_by_user_id`.

## Backend API (основные endpoint’ы)

Маршруты регистрируются в `irtran_server-main/src/modules/documents/documentReviewRoutes.js`.

- `POST /api/document-review/submit`
  - отправка документа студентом;
  - проверяет, можно ли повторно отправить документ;
  - вычисляет преподавателя по группам Keycloak.

- `GET /api/document-review/my`
  - список проверок текущего студента.

- `GET /api/document-review/teacher/submissions`
  - список отправок для преподавателя/админа;
  - фильтр по `status`.

- `GET /api/document-review/teacher/submissions/:id`
  - карточка конкретной отправки.

- `GET /api/document-review/teacher/submissions/:id/audit-trail`
  - история всех версий/оценок по документу студента.

- `GET /api/document-review/teacher/metrics`
  - мини-метрики:
    - `pendingCount`
    - `reviewedCount`
    - `acceptedCount`
    - `rejectedCount`
    - `avgReviewHours`

- `POST /api/document-review/teacher/submissions/:id/analyze`
  - алгоритмический анализ относительно эталона.

- `PATCH /api/document-review/teacher/submissions/:id/finalize`
  - финализация (или переоценка) проверки.

- `GET /api/document-review/templates`
- `PUT /api/document-review/templates/:documentType`
  - чтение/обновление эталонов.

## Frontend (основные компоненты)

- `IrTRAN-main/src/components/DocumentsListComponent.vue`
  - отправка документа на проверку;
  - подсказки по возможности повторной отправки.

- `IrTRAN-main/src/components/TeacherDocumentReviewComponent.vue`
  - рабочее место преподавателя:
    - список отправок;
    - мини-метрики;
    - комбинированный вердикт (алгоритм vs преподаватель);
    - чек-лист качества по секциям;
    - подсветка проблем с приоритетами;
    - audit trail;
    - финализация/переоценка.

- `IrTRAN-main/src/views/TeacherDocumentReviewManualView.vue`
  - readonly-страница документа для ручной проверки;
  - форматирование полей;
  - подсветка пропусков;
  - быстрые кнопки копирования замечаний преподавателя.

## Чек-лист качества и приоритеты проблем

В карточке проверки реализованы уровни:
- `critical` — критичные проблемы (красный);
- `warning` — предупреждения (жёлтый);
- `info` — информационные наблюдения (голубой).

Проблемы сортируются по приоритету и отображаются со счётчиками в заголовке блока.

## Комбинированный вердикт

В UI показывается:
- рекомендация алгоритма (`accepted/rejected`);
- решение преподавателя;
- признак совпадения/расхождения.

Это снижает риск “слепого” выставления оценки и помогает преподавателю аргументировать решение.

## Audit trail

Audit trail показывает:
- версию отправки;
- статус;
- решение/оценку;
- дату проверки.

Используется для прозрачности переоценок и контроля динамики исправлений.

## Инструкция на странице модуля

В модуле добавлена встроенная инструкция для преподавателя:
- сворачиваемый блок;
- состояние свёрнуто/развёрнуто сохраняется в `localStorage`.

## Ограничения MVP

- Глубокая проверка качества строк зависит от наличия расширенных структур в `payload_snapshot`.
- Для некоторых типов документов чек-лист пока общий, без детальной отраслевой разметки полей.
- Алгоритм сравнения работает по эталону типа документа, но не заменяет экспертную ручную проверку.
